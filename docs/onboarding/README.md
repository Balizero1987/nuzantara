# Onboarding Docs

Questa cartella contiene i documenti operativi per l’onboarding. Sono pensati per essere chiari, azionabili e misurabili, con una struttura modulare e collegata.

Contenuti
- `INDEX.md`: indice e guida di navigazione. Scopi, struttura, dipendenze, owner e come contribuire.
- `ORIENTATION_ONE_PAGER.md` (O1): mission/vision, mappa team/sistemi, accessi Day 0/Day 1, canali, lessico minimo.
- `NEW_JOINER_REPORT.md` (NJR): profilo, obiettivi 30/60/90, milestone, friction log, risk flags, learning plan.
- `CAPABILITY_MAP_DIGEST.md`: competenze chiave, livelli, mapping a ruoli/prodotti, gap e priorità formative.
- `FIRST_90_MINUTES.md` (F90): checklist tecnica e social/HR, mini‑deliverable, tempi stimati.
- `WEEKLY_DELTA_NEXT.md` (WDN): rituale delta/next, metriche onboarding, blocchi, piani di sblocco ed escalation.

Come richiedere una review
- Inviare il pacchetto a reviewer AI con il brief: `REVIEW_BRIEF.md`.
- Output atteso: unico file `REVIEW_YYYYMMDD_<MODEL>.md` con sezioni per ciascun documento, punteggi e priorità P0/P1/P2.
- SLA suggerito: R1 entro 24–48h, quindi merge R2 e sign‑off.

Workflow consigliato
- R0: bozza iniziale (skeleton completati)
- R1: feedback strutturato (rubrica+template)
- R2: consolidamento e refactor (link, checklist, metriche)
- Sign‑off: owner approva, si pubblica e si definisce cadence di update

Metriche e manutenzione
- KPI onboarding (esempi): time‑to‑first‑commit, time‑to‑access, numero blocchi risolti/settimana.
- Manutenzione: indicare owner e update cadence in `INDEX.md`; usare change log sintetico in coda a ogni file.
- Consistenza: mantenere glossario e naming coerenti, evitare duplicazioni tra file (preferire cross‑link).

Contatti

- Canale: `#onboarding-docs` per Q&A e triage
